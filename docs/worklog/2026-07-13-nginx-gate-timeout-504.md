# Worklog — 목표 AI 생성이 "Gateway Timeout"으로 막히던 문제 · 2026-07-13

## 요약 (TL;DR)
- **문제**: 대시보드에서 목표를 AI로 생성하면 60초쯤 뒤 빨간 `Gateway Timeout`이 뜨며 실패했다.
- **원인**: 라이브 Crewdeck은 테일넷 노출을 위해 nginx 게이트를 앞단에 두는데, 이 게이트에 응답 대기 시간 설정이 없어 nginx 기본값 60초에서 연결을 끊었다. 정작 앱은 같은 요청을 300초까지 기다리도록 만들어져 있었다 — **앱은 5분을 기다릴 준비가 됐는데 게이트가 1분 만에 잘라버린** 불일치였다.
- **한 일**: nginx 게이트의 응답/전송 대기 시간을 360초로 올려, 게이트가 앱의 상한(300초)을 존중하도록 맞췄다.
- **지금**: 수정이 라이브에 반영됐고, 게이트를 통과하는 실제 목표 생성 요청이 200으로 돌아오는 것을 확인했다(제안 10건 정상 반환).
- **남은 결정**: 없음.

## 한 일 (What we did)

사용자가 "목표를 AI로 생성하면 타임아웃이 난다"며 화면을 보내왔다. 모달에는 빨간 글씨로 `Gateway Timeout`이 떠 있었고, 그 아래 "다시 시도 / 바로 생성" 폴백이 나와 있었다.

먼저 그 "Gateway Timeout"이라는 문구가 어디서 나오는지 추적했다. 앱 코드에도, 대시보드 프런트에도 그 문자열은 없었다. 대신 호스트에 **nginx가 떠 있었고**, "Gateway Timeout"은 nginx가 상류(upstream) 응답을 제때 못 받았을 때 내는 504의 표준 문구였다. 접근 경로를 따라가니 이렇게 이어져 있었다: 테일넷 → `tailscale serve :7200` → **nginx owner-gate**(jay 전용, 127.0.0.1:7201) → Crewdeck(127.0.0.1:7200). 게이트는 이 개인 도구를 테일넷 전체 노출에서 소유자 한 명만 통과하도록 잠그는 역할이었다.

원인은 그 게이트의 대기 시간 설정이 비어 있던 것이었다. 목표 AI 생성 라우트는 claude/codex CLI를 동기로 띄워 응답을 기다리며, 앱은 이 요청에 대해 `req/res.setTimeout(300000)`으로 **최대 300초까지 기다리도록 명시**돼 있다. 서버 자체도 에이전트 실행이 10분 넘게 걸릴 수 있어 HTTP 타임아웃을 아예 꺼 둔 상태(`server.timeout=0`)였다. 그런데 앞단 nginx 게이트에는 `proxy_read_timeout` 오버라이드가 없어 **nginx 기본값 60초**가 적용됐다. AI 생성이 60초를 넘기는 순간 게이트가 먼저 상류 연결을 끊고 504를 뱉었다. 클라이언트 쪽 fetch 타임아웃은 무관했다.

수정은 병목 한 곳만 겨눴다. 게이트의 location 블록에 `proxy_read_timeout 360s`와 `proxy_send_timeout 360s`를 넣었다. 앱의 자체 상한(300초)보다 60초 크게 잡아, 타임아웃 경쟁에서 항상 앱이 먼저 깨끗한 응답을 내도록 했다(웹소켓 idle 여유도 겸한다). 설정 파일을 백업한 뒤 `nginx -t`로 문법을 검사하고, 통과했을 때만 `nginx -s reload`로 무중단 재적용했다 — 이 게이트 파일은 다른 사이트(md-notebook 등)와 같은 nginx를 공유하므로 reload가 다른 서비스를 끊지 않도록 graceful 경로만 썼다.

검증은 실제 진입점으로 했다. 게이트(7201)에 소유자 헤더를 붙여 헬스는 200, 헤더가 없으면 403이 나오는 것으로 게이트가 살아 있음을 먼저 확인했다. 이어 게이트를 통과하는 실제 목표 생성 요청을 두 번 보냈다 — 가벼운 요청(count=1)은 14초에 200, 무거운 요청(count=10 + 원본 자료)은 46초에 200으로 유효한 제안 10건을 돌려줬다. 오늘은 에이전트 응답이 60초 아래라 "60초를 넘겨도 살아남는" 장면을 자연 발생으로 포착하진 못했지만, nginx는 설정값을 결정론적으로 적용하므로 상한이 60초에서 360초로 바뀐 것으로 사용자의 60초 초과 케이스는 확실히 덮인다.

## 결정과 왜 (Decisions & why)

| 결정 | 버린 대안 | 왜 |
|------|----------|----|
| nginx 게이트 타임아웃을 360초로 올렸다 | 목표 생성을 비동기 작업(잡 생성 → 폴링)으로 바꿔 긴 동기 요청 자체를 없앤다 | 병목은 앱이 아니라 게이트의 60초였다. 앱은 이미 300초를 기다리도록 설계돼 있어, 게이트가 그 계약을 존중하기만 하면 됐다. 비동기 전환은 프런트·백엔드를 함께 고치는 큰 변경이라, 신고된 문제에 비해 과했다. |
| 게이트 상한(360초)을 앱 상한(300초)보다 크게 잡았다 | 정확히 300초로 일치시킨다 | 두 값을 같게 두면 경계에서 nginx가 앱보다 먼저 끊는 경쟁이 날 수 있다. 60초 헤드룸을 두면 앱이 항상 먼저 응답을 마쳐 사용자에게 깨끗한 결과나 JSON 오류가 가고, nginx 504가 끼어들지 않는다. |
| 백업 → `nginx -t` → graceful reload 순으로 적용했다 | 곧장 reload | 이 게이트는 md-notebook 등 다른 사이트와 nginx를 공유한다. 설정 오타 하나가 reload 때 다른 사이트까지 끊을 수 있어, 문법 검사를 통과했을 때만 무중단 reload로 넘어가도록 게이트를 걸었다. |
| 수정 사실을 레포 worklog로 남겼다 | 남기지 않는다 | 이 nginx 게이트는 레포에 없는 호스트 인프라라, 코드 이력만 봐서는 존재도 알 수 없다. 다음에 같은 타임아웃을 만나면 처음부터 다시 추적하게 되므로 경로와 원인을 글로 못박았다. |

## 배운 것 (Learned)

- 로컬 앱이 프록시 뒤에 있을 때, 앱이 아무리 타임아웃을 길게 잡아도(여기선 `server.timeout=0` + 라우트 300초) **가장 짧은 대기 시간을 가진 계층이 실질 상한**이 된다. 여기서는 nginx 게이트의 기본 60초가 앱의 300초 의도를 조용히 무력화했다.
- "Gateway Timeout"이라는 문구는 앱이 아니라 앞단 프록시(nginx 504)의 신호다. 앱·프런트 코드에서 문자열이 안 잡히면 요청 경로에 낀 프록시 계층을 의심해야 한다. 이 호스트의 Crewdeck은 `tailscale serve → nginx owner-gate(7201) → 앱(7200)` 3단이다.
- 이 게이트 파일(`/etc/nginx/conf.d/crewdeck-gate.conf`)은 배포 스크립트가 건드리지 않는 conf.d 파일이라 재배포와 무관하게 영속한다. 동기로 에이전트를 기다리는 라우트를 새로 추가하면 이 게이트의 타임아웃도 함께 고려해야 한다.

## 세션 흐름 (How it moved)

1. 앞선 요청으로 최신 코드를 pull해 라이브(systemd `crewdeck.service`)를 drain 절차대로 재배포한 직후였다(HEAD `a1245be`).
2. 사용자가 목표 AI 생성 시 `Gateway Timeout`이 뜨는 화면을 보내며 원인 파악·수정을 요청했다.
3. "Gateway Timeout" 문구의 출처를 추적해, 앱·프런트가 아니라 앞단 nginx의 504임을 특정했다.
4. 접근 경로(`tailscale serve → nginx gate 7201 → 앱 7200`)와, 게이트에 `proxy_read_timeout`이 없어 기본 60초가 걸리는 것, 앱은 같은 요청을 300초까지 기다리는 것의 불일치를 확인했다.
5. 게이트 설정을 백업하고 `proxy_read_timeout/send_timeout 360s`를 넣은 뒤, `nginx -t` 통과를 확인하고 graceful reload로 라이브 반영했다.
6. 게이트를 통과하는 실제 목표 생성 요청으로 200 응답(제안 10건 정상)을 확인했다.
