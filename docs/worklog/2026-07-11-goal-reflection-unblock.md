# Worklog — 다 만든 결과물이 반영되지 못하던 길목을 뚫은 날 · 2026-07-11

## 요약 (TL;DR)
- **문제**: 앱이 스스로 완성한 작업 묶음 세 개가, 다 만들어 놓고도 실제 프로젝트에 "반영"되지 못한 채 막혀 있었다.
- **한 일**: 반영을 가로막던 숨은 결함을 찾아 고치고, 막혀 있던 세 묶음을 모두 반영했다.
- **지금**: 반영 경로가 정상 작동한다. 덤으로, 앞으로 만들어지는 커밋 이름 앞에 종류 표시(새 기능·버그 수정 등)가 자동으로 붙는다. 상시 서비스에 반영하고 검증까지 마쳤다.
- **남은 결정**: 없음.

## 한 일 (What we did)
완성된 작업 묶음이 왜 반영되지 않는지부터 봤다. 세 묶음 모두 같은 이유로 막혀 있었다. 반영 직전에 작업물을 임시 저장하는 단계가 실패하고 있었던 것이다. 원인은 파일을 저장 목록에 담는 명령이, 도구가 만든 임시 폴더를 걸러내려다 오히려 오류를 내는 데 있었다. 최상위 폴더만 걸러내고 하위에 숨은 같은 이름의 폴더는 못 걸러 명령 전체가 죽었다.

이건 어제 테스트에서 한 번 격리했던 문제의 실제 서비스 버전이었다. 그때는 테스트만 손봤는데, 같은 뿌리가 반영 경로에도 살아 있었다. 걸러내기를 아예 빼니 명령이 저장 목록을 알아서 정리하며 깨끗하게 통과했다. 이 수정을 상시 서비스에 반영했다.

세 묶음 중 둘은 앱이 스스로 충돌을 풀어 반영됐다. 세 번째는 규모가 너무 커서(약 3900줄) 자동 해결이 15분 제한을 넘겨 실패했다. 다행히 작업물은 임시 보관함에 안전하게 남아 있었다. 별도 작업자에게 맡겨, 이 묶음의 의도와 그사이 본체가 앞서간 변경을 **양쪽 다 살리는** 방식으로 손수 합치게 했다. 합친 결과를 독립적으로 다시 검증한 뒤 반영했다.

마지막으로, 사용자가 아쉬워한 부분을 개선했다. 이 앱이 만드는 커밋에는 종류 표시가 없었는데, 이미 돌고 있는 요약 생성 단계에 종류 한 단어를 함께 뽑도록 얹어 커밋 이름 앞에 자동으로 붙게 했다.

## 결정과 왜 (Decisions & why)
| 결정 | 버린 대안 | 왜 |
|------|----------|----|
| 걸러내기 명령을 확장하지 않고 아예 제거 | 하위 폴더까지 걸러내도록 패턴을 늘리기 | 숨은 폴더가 어느 깊이에 생길지 알 수 없어 패턴 확장은 끝이 없다. 저장 명령은 원래 무시 규칙을 알아서 존중하므로, 걸러내기 자체가 불필요하고 오히려 해로웠다. |
| 세 번째 묶음은 작업자에게 수동 병합 위임 | 자동 재시도, 또는 보류 | 자동은 이미 15분 제한으로 실패했고 재시도해도 같은 규모라 또 실패할 가능성이 높았다. 작업물이 안전하게 보관돼 서두를 필요가 없었으므로, 대규모 의미 충돌은 사람 손으로 확실하게 푸는 편이 옳았다. |
| 커밋 종류 표시를 기존 요약 단계에 얹음 | 별도 AI 호출을 새로 두거나, 제목 키워드로 추측 | 요약을 만드는 값싼 호출이 이미 매번 돈다. 거기에 한 단어를 더 뽑으면 추가 비용이 없고, 키워드 추측보다 정확하다. |
| 반영된 커밋의 실시간 알림 코드를 제거된 채로 병합 | 옛 알림 코드를 되살려 남김 | 세 번째 묶음이 그 알림을 별도의 안전한 발행 창구로 이미 옮겨 두었다. 되살리면 같은 알림이 두 번 나간다. |

## 배운 것 (Learned)
- 테스트에서 격리한 환경 취약성은 실제 서비스 코드에도 같은 뿌리로 살아 있을 수 있다. 테스트만 고치고 끝내면 본체의 같은 결함을 놓친다.
- 큐를 멈춰도 이미 진행 중인 작업 세션은 멈추지 않는다. "개입해도 안전한 시점"을 판단할 때 큐 상태만으로는 부족하고 실제 프로세스를 봐야 한다.
- 자동 충돌 해결은 규모에 한계가 있다. 변경이 아주 크고 의미가 얽히면 사람이나 별도 작업자의 손이 필요하며, 그때 원본을 안전히 보관하는 장치가 있으면 서두르지 않고 제대로 풀 수 있다.

## 세션 흐름 (How it moved)
1. 완성된 세 묶음이 "반영 차단" 상태로 막혀 있음을 발견했다.
2. 반영 직전 임시 저장 명령이 도구 폴더를 걸러내다 죽는 것을 원인으로 특정했다.
3. 걸러내기를 제거해 고치고, 상시 서비스에 다시 빌드해 반영했다.
4. 두 묶음은 자동으로 반영됐고, 세 번째는 규모가 커서 자동 해결이 시간 제한을 넘겨 막혔다.
5. 작업물이 임시 보관함에 안전함을 확인한 뒤, 작업자에게 수동 병합을 맡겼다.
6. 합친 결과를 독립 검증하고 반영했으며, 원격 저장소로 올리는 과정의 충돌도 양쪽을 살려 풀었다.
7. 커밋 이름에 종류 표시를 자동으로 붙이는 개선을 구현해 반영했다.

## 남긴 것 (Left open)
- 반영을 마친 두 번째 묶음의 임시 작업 폴더가 정리되지 않고 남아 있다. 동작에 해는 없으나 정리 훅에 빠진 구석이 있어 보인다.
- 커밋 이름의 종류 표시는 **다음 묶음이 완료·반영될 때부터** 실제로 붙는다. 실제 반영 결과는 아직 관찰하지 않았으므로 미검증이다.

## 결과·검증 (Outcome)
막혀 있던 세 묶음이 모두 반영됐다. 반영 경로를 고친 뒤 서버 타입 검사와 핵심 테스트(마흔두 건)가 통과했고, 수동으로 합친 묶음은 별도 작업자가 494개 테스트를 통과시킨 것을 독립적으로 다시 확인했다. 이번 작업의 커밋들을 원격 저장소에 모두 올렸다. 다만 커밋 종류 표시가 실제 커밋에 붙는지는 다음 묶음이 반영돼야 확인되므로 **미검증**으로 남는다.
