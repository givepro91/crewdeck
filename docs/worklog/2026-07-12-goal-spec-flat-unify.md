# Worklog — Goal 기획서 생성을 실제 쓰이는 형식으로 통일 · 2026-07-12

## 요약 (TL;DR)
- **문제**: AI가 목표마다 자동으로 만들어 주는 "작업 기획서"가 얼마나 쓸모 있게 뽑히는지 점검해 달라는 요청에서 시작했다. 점검해 보니 기획서 자체는 잘 만들어지는데, 만든 내용의 절반이 실제 작업으로 전달되기 직전에 버려지거나 엉뚱한 항목으로 뒤바뀌고 있었다.
- **한 일**: 기획서를 만드는 부분이 처음부터 실제로 쓰이는 형식 그대로 뽑도록 고쳤다.
- **지금**: 이제 "이번 작업에서 제외할 범위"와 "무엇으로 검증할지"가 제대로 채워진다. 실제 서비스에 새 목표를 하나 넣어 기획서가 올바르게 나오는 것까지 확인했다.
- **남은 결정**: 없음. 다만 자동 진행 모드가 새 목표마다 사람 승인을 기다리는 것은 원래 의도된 설계이니 그대로 받아들이면 된다.

## 한 일 (What we did)
처음엔 단순 점검이었다. 기획서를 만드는 프롬프트와 실제 운영 데이터베이스에 쌓인 기획서 18건을 함께 읽었다. 생성물의 품질은 신뢰할 만했다. 문제는 그다음이었다.

만들어진 기획서는 실행 단계로 넘어가기 직전에 다섯 개 필드로 납작하게 옮겨진다. 이 옮기는 과정이 손실적이었다. "제외 범위" 필드는 항상 빈칸으로 채워졌고, "검증 방법" 필드에는 검증 방법 대신 아키텍처 조언이 들어갔으며, 기능별 우선순위는 통째로 사라졌다. 더 파보니 원본의 풍부한 필드를 실제로 읽는 코드는 어디에도 없었다. 납작한 형식이 실행의 유일한 계약인데, 생성기만 낡은 풍부한 형식을 계속 뱉고 있었던 것이다.

고침 자체는 함수 하나였다. 생성기가 납작한 형식을 직접 뱉도록 프롬프트와 저장 로직을 바꿨다. 이미 잘 돌던 다른 경로(사용자가 기획서를 다듬는 refine 경로)가 정답 형태를 보여주고 있어서, 그 형태를 그대로 따랐다.

가장 중요한 부분은 검증이었다. 내 판단만으로 결정하지 않기 위해 독립 검증자 둘을 붙였고, 이들이 내 사실은 확인했지만 판단의 왜곡 두 가지를 잡아냈다. 마지막엔 실제 운영 서비스에 목표를 하나 넣어 진짜 기획서가 올바르게 나오는 것까지 확인했다.

## 결정과 왜 (Decisions & why)
| 결정 | 버린 대안 | 왜 |
|------|----------|----|
| 생성기가 납작한 형식(`flat`, 다섯 필드로 정리한 실행용 명세)을 직접 뽑게 함 | 원본의 풍부한 정보를 실행 단계로 배선해 살림 | 그 풍부한 정보를 실제로 읽는 곳이 하나도 없었다. 있지도 않은 소비자를 위해 배선하는 셈이라 무의미하다. 사용자가 기획서를 다듬는 경로는 이미 납작한 형식을 직접 뽑고 있어 정답 형태가 이미 존재했다. |
| 낡은 풍부한 형식을 담던 테이블은 지우지 않고 최소만 채움 | 테이블 삭제 또는 스키마 변경 | 운영 데이터 18건과 상태 표시, 하위 호환 경로가 그 테이블에 걸려 있다. 스키마를 바꾸면 위험이 크므로, 지우지 않고 최소 호환만 남기는 쪽이 안전하다. |
| 자동 진행이 기획서 승인 대기에서 멈추는 동작은 건드리지 않음 | 자동 승인으로 무중단 복원 | 설계 문서가 수동 실행과 자동 진행을 포함한 네 실행 경로 모두 승인 없이는 실행 금지라고 명시했다. 이것은 버그가 아니라 의도된 관문이므로 이번 범위 밖이다. |
| 단위 테스트가 아니라 실제 라이브 목표로 최종 검증 | 자동 테스트 통과로 갈음 | 기존 테스트는 생성기를 가짜로 바꿔치기해서 돌린다. 그래서 내가 고친 로직에는 실제 커버리지가 없었고, 진짜 진입점으로 한 번은 돌려봐야 했다. |

## 배운 것 (Learned)
- 구현보다 먼저 작성된 설계 문서를 감사 계약으로 쓸 수 있다. 그 문서의 최소 합격 조건 일곱 개를 코드와 대조하니, 어려운 부분인 승인과 동시성 기계장치는 견고하고 결함은 생성 이음매 한 곳에만 몰려 있음이 드러났다.
- 독립 검증은 사실 오류만이 아니라 판단의 왜곡도 잡는다. 내가 정리한 사실 여덟 개는 모두 맞았지만, 심각도를 과장한 두 건과 가장 크게 체감될 결과 하나를 놓친 것을 검증자가 잡아냈다.
- 같은 데이터가 두 형식으로 공존할 때는, 한쪽을 고치기 전에 실제로 어느 쪽이 소비되는지부터 확인해야 한다. 소비되지 않는 쪽을 살리려는 노력은 대개 헛수고다.

## 세션 흐름 (How it moved)
1. 기획서 생성 품질을 점검해 달라는 요청으로 시작해, 생성 프롬프트와 운영 데이터베이스의 기획서 18건을 함께 읽었다.
2. 생성물은 우수하나, 만든 내용을 실행 직전에 다섯 필드로 옮기는 과정에서 정보가 손실됨을 발견했다.
3. 더 파보니 풍부한 형식의 필드는 읽는 곳이 없었고, 납작한 형식이 유일한 실행 계약이었으며, 이 신형 흐름은 서비스에서 아직 한 번도 돌지 않아 결함이 잠복 상태였다.
4. 사실을 검증하는 쪽과 비평하는 쪽으로 독립 검증자 둘을 붙였다. 사실 여덟 개는 모두 확인됐고, 심각도 과장과 자동 진행 정지 누락이 교정됐다. 설계 문서로 그 정지가 의도임을 확정했다.
5. 생성기 함수 하나를 납작한 형식 직접 출력으로 고치고, 타입 검사와 전체 자동 테스트 501개를 통과시켰다. 이어 실제 모델로 한 번 생성해 보고, 고친 저장 로직을 격리된 환경에서 따로 실행해 확인했다.
6. 독립 코드 리뷰어가 승인했고, 커밋한 뒤 무중단 절차로 라이브에 반영했다.
7. 실제 서비스에 목표 하나를 넣어 기획서가 올바르게 나오는 것을 확인하고, 흔적을 남기지 않도록 그 목표를 삭제했다.

## 남긴 것 (Left open)
- 사용자가 기획서를 다듬는 경로에도 이번에 넣은 배열 정리 로직을 똑같이 적용해 두 경로의 일관성을 맞추는 일은 선택적 후속으로 남겼다. 이번 요청 범위 밖이라 손대지 않았다.

## 결과·검증 (Outcome)
커밋 하나(`58a3efd`)로 반영하고 라이브 서비스를 무중단 절차로 재시작했다. 자동 채점 도구(`/gate`)는 돌리지 않았으므로, 여기 적는 검증은 모두 내가 직접 관찰한 것이다.

실제 운영 서비스에 새 목표를 넣어 실제 에이전트가 기획서를 생성하게 했다. 이전엔 항상 비어 있던 "제외 범위"가 실제 내용으로 채워졌고, 아키텍처 조언으로 뒤바뀌던 "검증 방법"이 타입 검사와 자동 테스트, 브라우저 확인 같은 실제 검증 절차로 나왔다. 그 목표가 승인 대기 상태에서 멈춰 작업이 저절로 진행되지 않는 것과, 테스트 목표를 지운 뒤 데이터에 흔적이 남지 않는 것까지 확인했다.
