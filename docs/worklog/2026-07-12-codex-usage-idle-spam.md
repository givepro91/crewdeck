# Worklog — Codex 사용량 게이지 + 활동 로그 폭주 수정 · 2026-07-12

## 요약 (TL;DR)
- **문제**: 화면 상단 정보 바에서 Claude 사용량은 보였지만, ChatGPT 구독으로 도는 코딩 에이전트(Codex)를 얼마나 썼는지는 알 수가 없었다. 게다가 "최근 활동" 목록에 똑같은 "대기 중" 메시지가 1초에 한 줄씩 쌓여 진짜 활동을 덮어버렸다.
- **한 일**: Codex 사용량을 Claude와 똑같은 모양의 막대(게이지)로 상단 바에 띄우고, 활동 목록이 도배되던 원인을 없앴다.
- **지금**: 두 변경 모두 상시 운영 중인 서비스에 반영됐고, 화면 캡처와 테스트로 동작을 확인했다.
- **남은 결정**: 없음. 선택적 후속 정리 항목만 남겨 두었다.

## 한 일 (What we did)

상단 정보 바는 Claude 계정의 5시간·주간 사용량을 막대로 보여주고 있었지만 Codex 쪽은 그 자리에 아무것도 없었다. 원인은 Codex를 비대화 모드로 실행할 때 표준 출력에 사용량 한도가 실려 나오지 않기 때문이었다. 그런데 Codex는 매 세션마다 남기는 기록 파일(rollout)에는 5시간·주간 사용률을 그대로 적어 둔다는 걸 실측으로 확인했다. 이 값은 Claude가 쓰는 소스와 구조가 사실상 같았다.

그래서 가장 최근 기록 파일에서 사용량 한도만 읽어 오는 서버 함수 하나와, 그걸 내려 주는 주소(`/api/codex-status`)를 새로 만들었다. 화면에서는 Claude 막대를 그리던 로직을 그대로 재사용해 Codex 막대를 나란히 붙였다. 요금제 이름(`pro`)과 한도가 다시 차오르는 시각도 함께 보여 준다. 왼쪽은 이 도구가 오늘 한 일(활성 수, 비용, 토큰, 세션 수), 오른쪽은 공급자별 남은 사용량으로 성격을 갈랐다.

작업 도중 두 번째 문제가 들어왔다. "최근 활동" 목록에 특정 에이전트의 "대기 중" 메시지가 같은 시각에 수십 줄씩 반복돼 쌓이는 현상이었다. 추적해 보니 스케줄러(작업을 자동 배분하는 루프)가 1초마다 도는 점검에서, 승인 대기 중인 목표가 있으면 매번 에이전트를 "쉬는 중" 상태로 되돌리며 그 사실을 화면에 밀어 보내고 있었다. 실제로는 상태가 바뀐 게 아닌데도 매초 같은 신호를 보낸 것이다.

수정은 실제로 상태가 바뀔 때만 신호를 보내도록 조건을 하나 더 건 것이다. 이미 쉬는 중이면 아무것도 하지 않는다. 이 동작을 못 박기 위해 회귀 테스트를 새로 추가했다.

## 배운 것 (Learned)

- Codex는 비대화 실행의 표준 출력에는 사용량 한도를 주지 않지만, 세션 기록 파일에는 5시간·주간 사용률을 항상 남긴다. 이 소스는 Claude가 쓰는 사용량 파일과 구조가 같아, 사용량 표시를 공급자 대칭으로 맞출 때 재사용할 수 있다.
- 상태를 화면에 밀어 보내는 코드가 "무조건 보낸다"는 것 자체는 문제가 아니다. 문제는 그 코드가 **1초마다 도는 점검 루프의 마무리 블록에 있어, 상태 변화 없이 매초 재실행될 때** 생긴다. 같은 종류의 신호 발생 지점 26곳을 전수 점검했지만 이 조합에 해당하는 곳은 스케줄러 한 곳뿐이었다.

## 세션 흐름 (How it moved)

1. 상단 바 현재 구성을 뜯어보고 Codex 사용량이 비어 있는 원인을 짚었다.
2. Codex가 사용량 한도를 세션 기록 파일에 남긴다는 사실을 최소 실행으로 실측했다.
3. 레이아웃 방향을 사용자와 합의했다(대칭 막대).
4. 격리 작업공간에서 구현하고, 별도 포트와 데이터로 띄운 인스턴스에서 화면과 응답을 검증했다.
5. 도중 들어온 활동 폭주 문제의 원인을 추적하고, 상태 전이 조건을 걸어 고친 뒤 회귀 테스트로 실증했다.
6. 진행 중 세션을 비우는 절차를 밟아 운영 서비스에 반영하고 다시 확인했다.
7. 비슷한 신호 발생 지점 전체를 점검해 추가 수정이 필요 없음을 확인했다.

## 남긴 것 (Left open)

- 화면 쪽 활동 목록에 "직전 항목과 같으면 건너뛰기" 중복 방지를 덧대는 보강(선택). 서버에서 원인을 이미 끊었으므로 필수는 아니다.
- 활동 목록 시각 표시가 초 단위를 생략해, 짧은 간격의 항목이 같은 시각처럼 보이는 점(디버깅 편의용 개선 후보).
- 태스크 실행 종료 처리에서, 드문 중복 실행 경합 시 다른 실행의 에이전트 상태를 잠깐 "쉬는 중"으로 덮을 여지가 있다. 활동 폭주와는 무관하고 빈도가 극히 낮아 이번 범위에서 제외했다.

## 결과·검증 (Outcome)

서버와 화면 타입 검사를 모두 통과했고, 빠른 테스트는 회귀 테스트 1개를 새로 더해 502개가 통과한다. 활동 폭주 수정은 가짜 통과가 아님을 직접 못 박았다. 가드를 일부러 빼면 5초 동안 "쉬는 중" 신호가 6번 나갔고, 가드를 되돌리면 실제 전이 1번으로 줄었다.

Codex 사용량 읽기 함수는 실제 기록 파일에서 5시간 5%, 주간 23%, 요금제 `pro`를 정확히 뽑아냈다. 새 주소의 응답도 확인했다. 운영 서비스에는 진행 중 세션을 비우는 절차를 밟아 반영했다. 반영 뒤 상단 바 화면 캡처로 Codex 막대가 Claude와 나란히 뜨는 것을 확인했고, 서비스 상태 점검도 정상이었다.
