# Crewdeck Terminal Workspace 제품 감사

## 범위

- 화면: Orca형 Workspace의 로컬 터미널, 목표·태스크, 에이전트 조직, 연결 가이드
- 사용자 목표: 터미널에서 Claude/Codex와 작업할 때 Crewdeck 오케스트레이션이 어떻게 관여하는지 즉시 이해하고 실행하기

## 결론

기술 연결은 존재하지만 제품 경험에서는 드러나지 않는다. 현재 화면은 `터미널 + Crewdeck 대시보드`가 나란히 놓인 상태이며, 터미널 행동과 목표·태스크 상태 변화 사이의 인과관계가 보이지 않는다. Crewdeck의 제품 핵심은 웹 터미널이 아니라 **모든 터미널 세션을 목표·에이전트·태스크·품질 판정에 귀속시키는 작업 제어면**이어야 한다.

## 흐름 감사

1. Workspace 진입 — 부분 양호
   - 프로젝트, goal worktree, 에이전트, 목표, 태스크를 한 화면에 배치한다.
   - `Crewdeck 연결됨`은 연결 유무만 말하고 무엇이 연결됐는지 설명하지 않는다.
2. 터미널 작업 시작 — 취약
   - 빈 프롬프트에서 사용자가 무엇을 입력해야 하는지, 현재 어떤 태스크를 맡았는지 알 수 없다.
   - Claude/Codex 버튼은 실행 수단이지만 목표를 실행하는 명확한 시작점으로 보이지 않는다.
3. 연결 방식 이해 — 취약
   - 도움말에는 MCP와 동기화 명령이 설명되어 있으나 핵심 가치가 `?` 뒤에 숨겨져 있다.
   - `crewdeck_get_context` 같은 구현 용어는 사용자가 원하는 “다음 태스크를 시작한다”는 행동 언어가 아니다.
4. 진행 관찰 — 부분 양호
   - 오른쪽 패널은 목표와 태스크를 보여주지만 정적 목록에 가깝다.
   - 어느 터미널 명령이나 Claude 행동이 어떤 태스크를 변경했는지 연결선과 활동 근거가 없다.
5. 완료·판정·다음 작업 — 확인 불가
   - 현재 화면에서는 작업 시작부터 Quality Gate, 완료, 다음 태스크 이동까지의 연속 경험이 보이지 않는다.

## 제품 방향

### 제품 정의

`Crewdeck = goal-bound terminal orchestration`  
Orca의 workspace/worktree/PTY를 기반으로 하되, Crewdeck은 각 세션을 `Workspace → Goal → Agent → Active Task → Evidence → Verdict`에 귀속시킨다.

### 기본 경험

1. Workspace를 열면 터미널 상단에 현재 귀속 상태를 고정 표시한다.
2. 태스크가 없으면 `다음 태스크 가져오기`를 가장 강한 CTA로 제공한다.
3. CTA를 누르면 담당 에이전트와 태스크가 세션에 binding되고 Claude/Codex가 해당 context로 시작한다.
4. 파일 변경, 명령 실행, task status, evaluator 판정을 이벤트로 묶어 오른쪽 `현재 작업` 패널에 실시간 표시한다.
5. PASS면 다음 태스크를 제안하고 FAIL이면 동일 세션에 수정 요구를 반환한다.

## 우선순위

1. Session Binding Bar: Goal, Agent, Active Task, 연결 상태, 다음 행동
2. `다음 태스크 가져오기` / `현재 태스크로 Claude 시작` 단일 진입점
3. 터미널 이벤트와 태스크 상태를 연결하는 실시간 Activity Rail
4. 완료 요청 → Quality Gate → PASS/FAIL → 다음 태스크의 명시적 루프
5. 중복된 에이전트·목표 목록을 줄이고 오른쪽 패널을 `현재 작업` 중심으로 재구성

## 접근성 관찰

- 상태 표시가 녹색과 작은 텍스트에 의존한다.
- `+`, `?`, `■`처럼 의미가 아이콘에만 있는 작은 조작부가 많다.
- 보조 텍스트 대비가 낮고 긴 한글 goal/branch 이름이 읽기 어렵다.
- 스크린샷만으로 키보드 순서, 스크린리더 이름, 200% 확대 동작은 확인할 수 없다.
